# 上下文工程：从无限信息到有限窗口

你有一个问题要解决。

相关知识可能有 100 万字。模型的 context window 是 20 万 token。

**你怎么决定哪些内容塞进去？**

这就是上下文工程的核心问题：**在有限的空间里，放入最有价值的信息。**

---

## 一个真实的困境

你在做一个代码助手，用户问：

```
这个项目的登录功能有 bug，帮我查查
```

你的选择：

| 选项 | 内容 | Token |
|------|------|-------|
| A | 把整个项目代码都塞进去 | 50 万 token，超了 |
| B | 只塞登录相关的文件 | 5 万 token，但可能漏掉依赖 |
| C | 先让模型"猜"需要哪些文件 | 可能猜错，多轮浪费时间 |
| D | 用 RAG 检索相关代码片段 | 1 万 token，但可能碎片化 |

**没有完美答案。每个选择都有代价。**

---

## 业界的几种思路

### 思路一：RAG 检索

**做法**：把代码库切成小块，建向量索引。用户提问时，检索 Top-K 相关片段。

**优点**：
- 简单直接
- 适用于任何规模的知识库

**缺点**：
- 碎片化：检索到的可能是"孤岛"，缺少上下文
- 检索不准：相似度 ≠ 相关性
- 难以处理"需要综合多份信息"的问题

**典型工具**：LangChain、LlamaIndex 的 RAG 模块

---

### 思路二：Agent 自主探索

**做法**：不给模型塞内容，给它"工具"——让它自己决定要看哪些文件。

```
模型: 我需要看登录相关的代码，列出文件
工具: 找到 5 个相关文件
模型: 打开 auth/login.ts
工具: 返回文件内容
模型: 这个文件引用了 session.ts，我也要看
工具: 返回 session.ts
...
```

**优点**：
- 模型自己判断"什么相关"
- 可以追溯依赖链
- 不会一次性塞爆 context

**缺点**：
- 多轮对话，延迟高
- 模型可能"走弯路"
- 对模型的推理能力要求高

**典型工具**：Claude Code、Cursor Agent、OpenHands

---

### 思路三：分层加载

**做法**：先给"目录"，模型需要时再深入。

```
第一层：项目结构 + 文件列表（2K token）
第二层：选中文件的函数签名（5K token）
第三层：完整文件内容（按需加载）
```

**优点**：
- 不会一开始就爆 context
- 模型有"全局视野"
- 按需深入，效率高

**缺点**：
- 需要预先构建索引/摘要
- 可能"目录"本身就很大
- 依赖结构化数据

**典型工具**：Trae、Aider 的 repo-map 模式

---

### 思路四：压缩 + 摘要

**做法**：把大段内容压缩成摘要，保留关键信息。

```
原始：5000 行代码（10 万 token）
摘要：50 行关键逻辑 + 函数签名（1 千 token）
```

**优点**：
- 大幅节省 context
- 可以塞入更多"广度"

**缺点**：
- 压缩 = 信息丢失
- 摘要质量决定效果
- 可能漏掉关键细节

**典型工具**：LLMLingua、Anthropic 的 Contextual Retrieval

---

## 一个关键洞察

这四种思路，**不是互斥的，而是不同场景下的权衡**：

| 场景 | 推荐思路 |
|------|----------|
| 知识库稳定、查询简单 | RAG |
| 需要探索、追溯依赖 | Agent 自主 |
| 项目大、需要全局视野 | 分层加载 |
| Context 紧张、精度要求低 | 压缩摘要 |

**真正的上下文工程能力 = 知道什么时候用什么策略。**

## 为什么这是 Agent 的分水岭？

想象两个 Agent：

**普通 Agent**：
```
用户: 帮我修这个 bug
Agent: 把整个项目塞进 context
结果: 超了，报错，或者答非所问
```

**优秀 Agent**：
```
用户: 帮我修这个 bug
Agent: 先看项目结构，定位相关模块
Agent: 打开关键文件，理解逻辑
Agent: 发现依赖，追溯上游
Agent: 找到根因，给出修复
```

**区别在哪？**

普通 Agent 会"无脑塞"。优秀 Agent 会"思考要什么"。

这个"思考要什么"的能力，本质上就是：**在有限 context 里，放入最有价值的信息。**

## 留给第四章的问题

当你给 Agent 装上"工具"（读文件、搜代码、跑测试），它会变得越来越强大。

但也会面临新的挑战：

- 工具调用的结果要不要保留？
- 50 轮对话后，context 怎么办？
- 如何让 Agent "记住"之前的发现？

**上下文管理，是 Agent 工程化的核心难题之一。**

下一章，我们会深入讨论这些问题——以及业界正在尝试的解决方案。

## 思考题

> 如果让你设计一个"能读懂整个代码库"的 AI 助手，你会怎么管理上下文？

考虑以下场景：
- 10 万行代码的项目
- 用户问了一个跨模块的问题
- 需要同时理解配置、业务逻辑、数据库 schema

你会选择哪种策略？还是组合使用？为什么？

这个问题没有标准答案，但思考它会让你理解"上下文工程"的真正难度。
