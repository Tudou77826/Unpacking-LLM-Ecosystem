# 上下文工程：从无限信息到有限窗口

你有一个问题要解决。

相关知识可能有 100 万字。模型的 context window 是 20 万 token。

**你怎么决定哪些内容塞进去？**

这就是上下文工程的核心问题：**在有限的空间里，放入最有价值的信息。**

---

## 一个真实的困境

你在做一个代码助手，用户问：

```
这个项目的登录功能有 bug，帮我查查
```

你的选择：

| 选项 | 内容 | Token |
|------|------|-------|
| A | 把整个项目代码都塞进去 | 50 万 token，超了 |
| B | 只塞登录相关的文件 | 5 万 token，但可能漏掉依赖 |
| C | 先让模型"猜"需要哪些文件 | 可能猜错，多轮浪费时间 |
| D | 用 RAG 检索相关代码片段 | 1 万 token，但可能碎片化 |

**没有完美答案。每个选择都有代价。**

---

## 业界的几种思路

### 思路一：RAG 检索

**做法**：把代码库切成小块，建向量索引。用户提问时，检索 Top-K 相关片段。

**优点**：
- 简单直接
- 适用于任何规模的知识库

**缺点**：
- 碎片化：检索到的可能是"孤岛"，缺少上下文
- 检索不准：相似度 ≠ 相关性
- 难以处理"需要综合多份信息"的问题

**典型工具**：LangChain、LlamaIndex 的 RAG 模块

---

### 思路二：Agent 自主探索

**做法**：不给模型塞内容，给它"工具"——让它自己决定要看哪些文件。

```
模型: 我需要看登录相关的代码，列出文件
工具: 找到 5 个相关文件
模型: 打开 auth/login.ts
工具: 返回文件内容
模型: 这个文件引用了 session.ts，我也要看
工具: 返回 session.ts
...
```

**优点**：
- 模型自己判断"什么相关"
- 可以追溯依赖链
- 不会一次性塞爆 context

**缺点**：
- 多轮对话，延迟高
- 模型可能"走弯路"
- 对模型的推理能力要求高

**典型工具**：Claude Code、Cursor Agent、OpenHands

---

### 思路三：分层加载

**做法**：先给"目录"，模型需要时再深入。

```
第一层：项目结构 + 文件列表（2K token）
第二层：选中文件的函数签名（5K token）
第三层：完整文件内容（按需加载）
```

**优点**：
- 不会一开始就爆 context
- 模型有"全局视野"
- 按需深入，效率高

**缺点**：
- 需要预先构建索引/摘要
- 可能"目录"本身就很大
- 依赖结构化数据

**典型工具**：Trae、Aider 的 repo-map 模式

---

### 思路四：压缩 + 摘要

**做法**：把大段内容压缩成摘要，保留关键信息。

```
原始：5000 行代码（10 万 token）
摘要：50 行关键逻辑 + 函数签名（1 千 token）
```

**优点**：
- 大幅节省 context
- 可以塞入更多"广度"

**缺点**：
- 压缩 = 信息丢失
- 摘要质量决定效果
- 可能漏掉关键细节

**典型工具**：LLMLingua、Anthropic 的 Contextual Retrieval

---

## 一个关键洞察

这四种思路，**不是互斥的，而是不同场景下的权衡**：

| 场景 | 推荐思路 |
|------|----------|
| 知识库稳定、查询简单 | RAG |
| 需要探索、追溯依赖 | Agent 自主 |
| 项目大、需要全局视野 | 分层加载 |
| Context 紧张、精度要求低 | 压缩摘要 |

**真正的上下文工程能力 = 知道什么时候用什么策略。**

## 好消息：你不需要从零搭建

看到这里，你可能在想：**这也太复杂了，我自己哪搞得定？**

好消息是：**成熟的 Agent 工具已经帮你做了 80% 的工作。**

### Claude Code 是怎么做的？

- 自动分析项目结构，知道哪些文件"可能相关"
- 按需加载，不会一股脑塞进去
- 对话太长时自动压缩历史
- 重要信息放在 context 的"黄金位置"

### Cursor / Trae 是怎么做的？

- 建立代码索引，快速定位相关片段
- 智能决定"读多少"而不是"全读"
- 多文件编辑时保持上下文一致性

### 意味着什么？

| 你需要操心的事 | 工具已经帮你做的事 |
|---------------|-------------------|
| 清楚描述你的问题 | 检索相关代码 |
| 验证结果对不对 | 上下文管理 |
| 给出精准的反馈 | 提示词结构化 |

**你不需要成为"上下文工程专家"，工具给你提供了"及格分"的体验。**

---

## 那为什么还要学这些？

两个原因：

**第一，理解边界**

工具不是万能的。当你发现：
- 模型"忽略"了你明确提到的文件
- 对话一长就"变傻"
- 复杂任务总是做不对

你知道可能是上下文的问题，而不是"模型不行"。

**第二，特殊情况需要手动干预**

- 超大型项目（100 万行+）
- 特殊领域的知识库
- 需要精细控制的场景

这时候，理解原理能帮你"调优"。
