# 上下文工程：为什么"塞进去 ≠ 被用到"

第三课我们提过"Lost in the Middle"——模型容易忽略中间的内容。

这一课，我们不讲原理（已经讲过了），讲**实践中怎么踩坑、怎么避坑**。

---

## 一个真实的翻车现场

某公司用 RAG 做智能客服：

```
知识库：5000 份文档，每份平均 3000 字
检索策略：Top-20 相关文档
每条对话：带上全部 20 份文档 + 10 轮历史
```

结果：模型回答质量奇差，经常"答非所问"。

**问题在哪？**

- 20 份文档 × 3000 字 = 60,000 字
- 10 轮对话历史 ≈ 5,000 字
- System Prompt + 格式指令 ≈ 1,000 字
- **总计：66,000+ 字塞进 context**

模型"看到"了，但没"注意到"。

---

## 上下文爆炸的三种来源

### 来源一：RAG 检索结果的堆叠

最常见的问题：

```python
# 检索 Top-K 相关文档
results = vector_db.query(query, top_k=20)  # 20 份文档

# 全部塞进去
context = "\n\n".join([doc.content for doc in results])
# 20 × 3000 字 = 60,000 字
```

**直觉：信息越多越好，给模型更多参考资料**

**现实：太多信息 = 模型找不到重点**

### 来源二：多轮对话的累积

```python
messages = [
    {"role": "user", "content": "问题1"},
    {"role": "assistant", "content": "回答1（500字）"},
    {"role": "user", "content": "问题2"},
    {"role": "assistant", "content": "回答2（800字）"},
    {"role": "user", "content": "问题3"},
    # ... 10 轮后
]
# 累积 5,000-10,000 字很正常
```

每一轮都带上全部历史，对话越长，context 越膨胀。

### 来源三：工具调用记录

Agent 场景更严重：

```
用户：帮我分析这个项目的代码结构
模型：我需要先看看目录 [调用 list_files]
工具返回：[500 个文件路径]
模型：看看 package.json [调用 read_file]
工具返回：[2000 字的配置文件]
模型：看看 src 目录 [调用 list_files]
工具返回：[300 个文件路径]
...
```

**5 轮工具调用，context 可能已经 10,000+ 字。**

---

## 实用建议：让内容真正被用到

### 原则一：检索要克制，Top-5 够用

| Top-K | 字数 | 效果 |
|-------|------|------|
| Top-3 | ~9,000 字 | 精准，但可能漏 |
| Top-5 | ~15,000 字 | **最佳平衡点** |
| Top-10 | ~30,000 字 | 开始有噪音 |
| Top-20 | ~60,000 字 | 模型迷失 |

**不是越多越好，5 份高质量文档 > 20 份低质量文档。**

### 原则二：重要信息放开头或结尾

```
[最重要的指令 / System Prompt]     ← 高注意力
[检索到的文档]                      ← 中等注意力
[对话历史]                          ← 中等注意力
[当前问题]                          ← 高注意力
```

**把"必须遵守"的规则放开头，把"关键问题"放结尾。**

### 原则三：对话历史要"压缩"，不是"全带"

糟糕的做法：
```python
# 每次带上全部历史
messages = all_previous_messages + [current_message]
```

好的做法：
```python
# 只带摘要 + 最近 3 轮
summary = summarize_conversation(all_previous_messages)
recent = all_previous_messages[-6:]  # 最近 3 轮（用户+助手）
messages = [
    {"role": "system", "content": f"对话摘要：{summary}"},
    *recent,
    current_message
]
```

**10 轮对话的摘要可能只有 500 字，比原文 10,000 字高效得多。**

### 原则四：警惕隐式消耗

你以为只塞了 10,000 字，实际上：

| 内容 | 你以为 | 实际 |
|------|--------|------|
| System Prompt | 0 | 500-1000 字 |
| 格式指令 | 0 | 200-500 字 |
| 思考过程（CoT）| 0 | 1000-3000 字 |
| 工具描述 | 0 | 500-2000 字 |
| **隐式消耗** | **0** | **2000-6500 字** |

**加上你显式塞的 10,000 字，实际可能 12,000-16,500 字。**

## 工具层面：Claude Code / Trae 怎么管

你可能会问：我用 Claude Code 或 Trae，需要关心这些吗？

**大部分时候不需要。这些工具已经内置了上下文管理：**

### Claude Code 的做法

1. **选择性加载文件**：不会把整个项目塞进去，只加载"相关"的文件
2. **自动摘要**：对话太长时，自动压缩历史
3. **层级结构**：目录结构 < 文件列表 < 文件内容，按需深入

### Trae 的做法

1. **智能检索**：用向量搜索找相关代码片段，不是全文
2. **动态窗口**：只保留最近 N 轮对话的完整记录
3. **后台索引**：代码库提前建好索引，查询时只取片段

### 什么时候你需要关心？

| 场景 | 需要关心吗 |
|------|-----------|
| 用 Claude Code/Trae 写代码 | 不需要，工具帮你管了 |
| 自己开发 RAG 应用 | **需要**，你要设计检索策略 |
| 自己开发 Agent | **需要**，你要管理工具调用记录 |
| 调试"模型不按指令执行" | **需要**，检查是否 context 太乱 |

---

## 总结

| 问题 | 解决 |
|------|------|
| 检索结果太多 | Top-5 够用，质量 > 数量 |
| 对话历史太长 | 压缩成摘要，只带最近几轮 |
| 重要内容被忽略 | 放开头或结尾 |
| 工具调用膨胀 | 定期清理，只保留关键结果 |

**核心认知：Context Window 是"容量"，不是"注意力"。塞进去 ≠ 被看到 ≠ 被用到。**

---

## 思考题

> 你的 Agent 执行了 20 步操作，每步都产生了工具返回，现在 context 快爆了，怎么办？

- 直接截断？可能丢掉关键信息
- 全部摘要？可能丢失细节
- 只保留"失败"的步骤？可能丢失成功经验

这个问题没有标准答案，但思考它会让你理解"上下文管理"为什么是 Agent 工程化的核心难题。
