# ❓ 解构 LLM：第一性原理核心问题清单 (The Socratic Checklist)

> **“提出一个好问题，比得到一个答案更重要。这份清单是我们探索 AI 底层逻辑的罗盘。”**

## 🧱 第一章：大模型的基石（LLM 底层原理）
*告别拟人化想象，用数学和工程的眼光拆解“智能”的假象与真相。*

1. **文字接龙的本质**：LLM 所谓的“懂了”，在数学上到底是在计算什么？（Next Token Prediction 到底是怎么一回事？）
2. **升维与降维**：为什么文字必须被转换成向量（Embedding）？在多维空间里，“苹果”和“手机”的距离是如何决定的？
3. **Transformer 核心**：什么是注意力机制（Attention）？它到底是在“注意”什么？如果没有注意力机制，模型会有什么致命缺陷？
4. **炼丹三部曲**：预训练（Pre-training）、微调（SFT）和对齐（RLHF）分别赋予了模型什么能力？如果只做预训练，模型会是什么样子？
5. **智能的涌现**：为什么参数量大到一定程度，模型会突然学会做算术题或翻译？（涌现能力的本质是什么？）
6. **幻觉（Hallucination）的必然性**：为什么说幻觉不是 LLM 的 bug，而是它的 feature（特性）？从概率生成的角度，如何解释幻觉的产生？
7. **记忆的局限**：模型自身的“知识”储存在哪里？它是像数据库一样精确存储的，还是像人类大脑一样模糊压缩的？

## 🗣️ 第二章：对话的艺术与干预（Prompt Engineering）
*当模型的权重（Weights）被冻结，我们如何通过输入的文本改变它的输出路径？*

1. **In-Context Learning（上下文学习）的奇迹**：为什么我在提示词里给了几个例子（Few-shot），它就能学会新任务？在这个过程中，模型的参数真的改变了吗？
2. **思维链（CoT）的解剖**：为什么加上“让我们一步步思考”，模型的逻辑能力就会暴涨？它在内部的计算图谱上发生了什么变化？
3. **注意力分配**：冗长的提示词为什么往往效果不好？提示词的先后顺序（比如把指令放在开头还是结尾）如何影响 Attention 的分配权重？
4. **提示词的生命周期**：提示词工程是一门“玄学”还是科学？随着大模型推理能力（如 OpenAI o系列）的进化，传统的 Prompt 技巧还有效吗？

## 📚 第三章：记忆外挂与数据投喂（Context Engineering & RAG）
*解决模型“没见过”、“记不住”和“乱编造”的终极工程手段。*

1. **RAG（检索增强生成）的第一性原理**：既然微调（Fine-tuning）可以注入知识，为什么我们还需要 RAG？微调和 RAG 在解决“知识盲区”时的本质区别是什么？
2. **向量检索的黑盒**：向量数据库是如何快速在百万条文本中找到最相关的段落的？（余弦相似度等算法简述）
3. **Lost in the Middle（迷失在中间）现象**：为什么把几十万字的文件塞进上下文窗口（Context Window），模型往往能记住开头和结尾，却忘掉中间的内容？
4. **分块（Chunking）的艺术**：在把长文档切片喂给模型时，切大了和切小了各有什么灾难性后果？如何科学地分块？
5. **召回与生成**：如果检索（Retrieval）到的知识就是错的，LLM 能否自我纠错？RAG 系统的瓶颈到底在检索端，还是在生成端？

## 🛠️ 第四章：赋予手眼与统一标准（Skill / Tool Calling & MCP）
*让模型从“纸上谈兵”走向“改变真实世界”。*

1. **Tool Calling 的魔术**：一个只能输出文本的模型，究竟是如何“点击”网页、“查询”数据库或“运行”Python 代码的？（解析 API 调用的 JSON 格式黑盒）
2. **ReAct（推理+行动）框架**：模型是如何判断“什么时候该调用工具”，以及“该调用哪个工具”的？
3. **插件生态的坟墓**：为什么 ChatGPT 早期的 Plugins 生态没有做起来？应用之间互相接入的痛点到底卡在哪里？
4. **破局者 MCP（模型上下文协议）**：
   * MCP 到底是个什么协议？它定义了哪些标准交互接口？
   * 为什么我们把 MCP 称为 AI 时代的“USB 接口”？它解决了开发者在集成 AI 时的什么噩梦？
   * 如果我想让 LLM 安全地读取我本地电脑上的隐私文件，通过 MCP 的架构是如何实现且保证安全的？

## 🛡️ 第五章：自治时代的“围栏”（Harness Engineering / 驾驭工程）
*当 Agent 能够独立编写十万行代码、自主调用财务 API 时，人类的最终护城河在哪里？*

1. **角色的演变**：为什么在 Agent 时代，软件工程师的核心工作不再是“实现功能”，而是“定义边界”和“建立约束”？
2. **幻觉的放大效应**：当单个 LLM 的幻觉率为 2% 时，让 5 个 Agent 串联工作，系统崩溃的概率会呈指数级上升吗？如何对抗这种系统性熵增？
3. **架构约束（Architectural Constraints）**：怎样在代码层面给 AI 戴上“紧箍咒”？（什么是结构化输出约束？如何利用 Linter 和沙盒隔离 AI 的破坏力？）
4. **闭环反馈机制（Feedback Loops）**：当 Agent 产出的代码出现 Bug，谁来报错？如何构建“模型审查模型（LLM-as-a-Judge）”的自动化纠错闭环？
5. **可解释性与监控（Observability）**：当一个黑盒（LLM）调用另一个黑盒（外部复杂系统）时，人类如何监控它的思考轨迹（Trace）以防它“走火入魔”？
6. **Human-in-the-loop（人在回路）的终极形态**：在高度自治的 AI 系统中，人类的干预点应该设计在哪里？是审批每一次 API 调用，还是只负责设定最终的 KPI 和底线？

通过这种“自问自答”的方式推进仓库建设，你的内容将拥有极强的**逻辑张力**和**极客深度**！
