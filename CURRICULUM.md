# 📚 解构 LLM：完整课程目录

> **课程定位**：面向程序员的 AI 编程公开课——不是教你"怎么用"，而是帮你搞懂"为什么是这样"。

---

## 🧱 第一章：大模型的基石（共 3 课时）

> **本章目标**：建立对 LLM 工作方式的直觉理解，揭示其"原生困境"，理解为什么后续会出现提示词工程、RAG、Tool Calling、Harness Engineering 等技术——它们都是为了解决这些困境。

---

### 第 1 课：LLM 的本质——概率预测与语义空间

**📝 课程大纲**
- **从扔硬币到语言模型**：概率分布的直觉
- **Token 与 Next Token Prediction**：
  - 文本世界的"原子"
  - "预测下一个词"的数学含义
  - Softmax、采样与 Temperature
- **Embedding 与语义空间**：
  - 为什么文字必须变成向量？
  - 在高维空间"看见"语义
  - "国王-男人+女人≈女王"的几何直觉
- **核心认知**：LLM 不是"思考"，而是"计算概率"

**🔓 揭开的疑惑**
- 为什么说 LLM 本质上是"高级自动补全"？
- 同样的问题为什么每次回答不一样？
- 为什么计算机能理解"苹果"和"香蕉"是类似的东西？
- Temperature=0 和 Temperature=1 到底差在哪？

**🚧 引出的困境**
- 概率生成的不确定性 → **可控性困境**
- 语义理解依赖上下文 → **上下文理解困境**

---

### 第 2 课：Transformer 与训练——能力的诞生

**📝 课程大纲**
- **为什么是 Transformer**：
  - RNN/LSTM 的瓶颈：顺序处理的代价
  - Self-Attention：让每个词"看见"所有词
  - 注意力机制的直觉与影响
- **训练的三部曲**：
  - 预训练：在海量文本上学习"预测下一个词"
  - 微调（SFT）：从"话痨"变成"助手"
  - 对齐（RLHF/DPO）：让模型"听话"
- **涌现与规模**：为什么大模型突然会了小模型不会的事
- **核心认知**：模型的"知识"压缩存储在参数中

**🔓 揭开的疑惑**
- 为什么 Transformer 能完全并行化？
- "注意力"到底在注意什么？
- 为什么只预训练的模型不像"助手"？
- 模型参数里到底"存"了什么？

**🚧 引出的困境**
- 知识在训练时固化 → **知识时效性困境**
- 知识压缩存储 → **知识精确性困境**

---

### 第 3 课：原生困境——LLM 的边界与解决之路

**📝 课程大纲**
- **困境一：指令遵循不稳定**（用户最先遇到的痛点）
  - 同样的提示词，结果可能不同
  - 复杂指令容易"漏项"或"跑偏"
  - → **解决方向**：提示词工程（第二章）

- **困境二：知识是"凝固"的**
  - 训练截止日期 = 知识截止日期
  - 私有数据、企业内部数据模型没见过
  - → **解决方向**：外挂知识（第三章）

- **困境三：上下文有限且会"迷失"**
  - Context Window 有长度限制
  - "Lost in the Middle"：塞进去 ≠ 被用到
  - → **解决方向**：上下文工程（第三章）

- **困境四：只会"说"，不会"做"**
  - 模型只能输出文本，不能执行操作
  - 无法访问数据库、调用 API、读写文件
  - → **解决方向**：Tool Calling & MCP（第四章）

- **困境五：幻觉是"特性"不是"bug"**
  - 概率生成必然伴随幻觉
  - 模型不知道自己在"编"
  - → **解决方向**：Harness Engineering（第五章）

- **全景图**：困境与解法的对应关系

**🔓 揭开的疑惑**
- 为什么模型会"一本正经地胡说八道"？
- 为什么不能把所有知识都"教"给模型？
- 为什么光靠提示词有时不管用？
- 为什么需要 RAG、Tool Calling、Harness 这一套组合拳？

**🎯 本章小结：技术演进的内在逻辑**

| 原生困境 | 解决技术 | 所在章节 |
|----------|----------|----------|
| 无法稳定遵循指令 | Prompt Engineering | 第二章 |
| 知识固化、无法更新 | RAG | 第三章 |
| 上下文有限、易迷失 | Context Engineering | 第三章 |
| 只能输出文本、无法行动 | Tool Calling / MCP | 第四章 |
| 行为不可控、幻觉难消除 | Harness Engineering | 第五章 |

### 第 4 课：提示词工程基础——从 Zero-shot 到 Few-shot

**📝 课程大纲**
- Zero-shot：零样本推理的原理与局限
- Few-shot：上下文学习（In-Context Learning）的机制
  - 为什么几个例子就能改变行为？
  - 例子的选择与排序策略
- System Prompt vs User Prompt：角色的威力
- 结构化提示：分隔符、模板与格式约束

**🔓 揭开的疑惑**
- 为什么给几个例子模型就"学会"了新任务？
- 模型参数变了吗？那"学习"发生在哪里？
- 例子给多少个最合适？越多越好吗？
- System Prompt 为什么比 User Prompt 更"听话"？

---

### 第 5 课：提示词技巧的兴衰——从"必备技能"到"内置能力"

**📝 课程大纲**
- 曾经的三件"神器"：思维链、角色扮演、格式约束
- 它们为什么有效？从第一性原理解释
  - 思维链：拆解任务，降低每步的概率难度
  - 角色扮演：激活特定知识域的参数区域
  - 格式约束：缩小输出空间，提高命中概率
- 它们为什么正在消失？
  - 推理模型：CoT 原生化（o1、R1、GLM-4.7、Kimi K2.5）
  - 意图理解增强：模型能从上下文推断角色
  - API 级别支持：JSON Mode 成标配
- 2026 年的核心能力：问题拆解 + 反馈设计

**🔓 揭开的疑惑**
- 为什么"提示词工程师"这个职位消失了？
- 推理模型和普通模型有什么本质区别？
- 当模型能自己思考，我们该干什么？

### 第 6 课：RAG——为什么模型需要"外挂硬盘"

**📝 课程大纲**
- LLM 的知识边界：训练数据的时间与范围限制
- 为什么不直接"把资料贴给模型"？
  - Context Window 限制
  - 成本与延迟考量
- RAG 的核心思路：检索 → 喂给模型 → 生成
- 向量检索的直觉：用"语义相似"找相关内容
- RAG 不是万能的：什么时候不管用？
- 微调 vs RAG vs 长上下文：怎么选？

**🔓 揭开的疑惑**
- 为什么不能把公司文档全部微调进模型？
- RAG 和"直接把资料贴给模型"有什么区别？
- 长上下文模型会取代 RAG 吗？
- 什么时候该用 RAG，什么时候不该用？

---

### 第 7 课：上下文工程——为什么"塞进去 ≠ 被用到"

**📝 课程大纲**
- Context Window：不是"越大越好"
- Lost in the Middle：模型"看"不到中间的内容
- 上下文爆炸：为什么会发生，怎么避免
  - 多轮对话的累积
  - 检索结果的堆叠
  - 工具调用记录的膨胀
- 实用建议：
  - 重要信息放开头或结尾
  - 定期"清理"对话历史
  - 警惕隐式上下文消耗
- 工具层面：Claude Code、Trae 怎么管理上下文

**🔓 揭开的疑惑**
- 为什么塞了几十万字，模型却答不准？
- 分块太大太小各有什么问题？
- 对话越长模型越"傻"，怎么回事？
- 我需要关心上下文管理吗？还是交给工具？
